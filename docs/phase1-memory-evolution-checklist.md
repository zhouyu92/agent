# Phase 1 Memory Evolution Checklist

这份清单只服务一个目标：把当前 memory evolution MVP 从“已经可用”推进到“规则更稳、行为更可解释、治理更方便”。

不在这一阶段处理：

- LangGraph Phase 2 结构迁移
- 新的生产存储
- 大规模 CLI 重做
- 工具调用或多 agent 编排

## 1. 规则收紧

- [ ] 明确 `add` 的触发条件，避免把一次性寒暄或弱信号写成长久记忆
- [ ] 明确 `reinforce` 的触发条件，避免“相似但已变化”的信息被错误加强
- [ ] 明确 `revise` 的触发条件，优先处理会覆盖旧偏好、旧事实、旧边界的输入
- [ ] 明确 `ignore` 的触发条件，给低价值输入稳定落到忽略路径
- [ ] 形成一组可读的规则样例，覆盖 preference / profile / factual memory

完成标准：

- 同一类输入在 classic / langgraph 路径下产生一致的演化结果
- 常见低价值输入不会再进入长期记忆

## 2. 冲突与过时场景测试

- [ ] 增加“用户偏好改变”测试，例如喜欢 A 改成喜欢 B
- [ ] 增加“事实修正”测试，例如时间、地点、身份信息被更新
- [ ] 增加“同义重复”测试，确保重复表述不会不断 add
- [ ] 增加“弱信号输入”测试，确保寒暄、口头禅、短噪声默认 ignore
- [ ] 增加“旧记忆 superseded 后不参与默认检索”的测试

完成标准：

- memory evolution 的关键分支都有直接测试覆盖
- 出现回归时能从测试名快速定位是哪类学习错误

## 3. 路由与学习边界

- [ ] 再检查 routing policy 和 memory evolution 的责任边界
- [ ] 明确“该不该学习”与“学了之后怎么处理”是两个层次的问题
- [ ] 避免把 routing 中的启发式判断重复塞进 evolution 决策
- [ ] 为 thread inspection 保留足够上下文，能看清一轮里是“没学”还是“学了但 ignore”

完成标准：

- 能清晰区分以下几种情况：
  - 本轮没有进入学习
  - 本轮进入学习但被 ignore
  - 本轮触发 reinforce
  - 本轮触发 revise

## 4. CLI 审计体验

- [ ] 检查 `/thread` 对 memory evolution 事件的可读性
- [ ] 检查 `/learning` 和 `/memories` 联合使用时，是否能快速回放某次学习结果
- [ ] 如确有必要，再补最小范围的筛选或格式优化
- [ ] 保持命令面尽量小，优先增强现有查询而不是新增很多命令

完成标准：

- 人工排查一条错误记忆时，不需要翻代码或查数据库
- 仅用现有 CLI 就能回答“这条记忆为什么会出现”

## 5. 回归验证

- [ ] 跑全量 `pytest`
- [ ] 针对 memory / store / cli / agent 路径做重点回归
- [ ] 进行一次手工对话验证，覆盖 add / reinforce / revise / ignore 四种结果

完成标准：

- 自动化测试通过
- 手工演示能观察到四种演化结果及其审计轨迹

## 推荐执行顺序

1. 先补测试，再微调规则
2. 再收紧 routing 和 evolution 的边界
3. 最后只做必要的 CLI 可读性优化

## 这一阶段的完成标志

当你能比较有把握地回答下面 4 个问题，就说明 Phase 1 基本完成了：

- 为什么这一轮被写成长期记忆？
- 为什么这一轮只是 reinforce 而不是 revise？
- 为什么这条旧记忆被 superseded？
- 为什么这轮对话没有学到任何长期信息？
